# Script Agent Documentation

This document details the complete implementation of the Script Agent (implemented in the codebase as the **Storyboard Agent**) within the Manima platform.

---

## 1. High-Level Purpose

The Script/Storyboard Agent serves as the creative writer and storyboard artist for the video generation pipeline.

- **Responsibilities:** It takes the high-level, bullet-point curriculum plan and transforms it into a concrete, scene-by-scene script. It dictates exactly what the voiceover (TTS) will say and pairs it with explicit visual instructions and animation beats.
- **Input Provider:** The Planner Agent provides the teaching sequence, and the Concept Classifier provides the visual strategy/metaphor.
- **Output Consumer:** The immediate consumer of the output is the **Human Review** node (where users approve the script via the frontend) and the **Scene JSON Generator** (which turns the visual instructions into a declarative schema).

---

## 2. Inputs

The Storyboard Agent reads the following state variables from the `AgentState`:

- **`video_id` (str):** For logging and database updates.
- **`video_title` (str):** The title of the video, generated by the Planner.
- **`scene_plans` (List[ScenePlan]):** The high-level curriculum sequence generated by the Planner.
- **`visualization_strategy` (str):** The domain category determined by the Concept Classifier (e.g., `"data_structure_tree"` or `"physics_motion"`).
- **`visual_metaphor` (str):** A thematic visual instruction determined by the Concept Classifier.

All these inputs originate from earlier nodes in the LangGraph workflow.

---

## 3. Prompt Construction

The LLM prompt is assembled dynamically in `app/agents/prompts/storyboard_prompts.py` using `create_storyboard_prompt()`.

**System Prompt (`STORYBOARD_SYSTEM_PROMPT`):**
The system prompt places heavy emphasis on the correlation between audio and visuals. It enforces rules such as:
- **Pedagogical Priority:** The viewer should understand the concept even if audio is muted.
- **No Empty Scenes:** Black screens are forbidden; every scene must contain educational objects.
- **Animation Quality:** Static scenes longer than 2 seconds are forbidden.
- **Algorithm Rules:** Algorithms must be animated step-by-step.
- It strictly enforces the output to be valid JSON.

**User Prompt Assembly:**
The `create_storyboard_prompt` function extracts the `scene_number`, `title`, and `key_concepts` from the `scene_plans` and formats them into a readable sequence.
```text
Video Title: [video_title]
Visualization Strategy: [visualization_strategy]
Visual Metaphor / Theme: [visual_metaphor]

Teaching Sequence:
- Scene 1: Introduction to X (Concepts: concept A, concept B)
- Scene 2: ...

Create a detailed storyboard following the system instructions...
```

---

## 4. Model Configuration

The Script LLM is instantiated via `create_llm("scripter", temperature=0.7)` from `app/services/llm_factory.py`.

- **Provider:** Configured via environment variables (defaults to Groq or OpenRouter).
- **Model Name:** Follows the "scripter" role configuration.
- **Temperature:** `0.7` (allows for creative, engaging narration while adhering to the structural rules).
- **Response Format:** Requests JSON in the prompt text.
- **Retry Configuration:** A `max_attempts = 3` loop is used to handle parsing or validation failures.

---

## 5. Script Output

The agent outputs a list of `StoryboardScene` objects (defined as a `TypedDict` in `state.py`), which update the `storyboards` field in the `AgentState`.

| Field | Type | Purpose | Example |
|-------|------|---------|---------|
| `scene_number` | `int` | Sequential ordering. | `1` |
| `goal` | `str` | The learning objective of the scene. | `"Introduce linear regression"` |
| `narration` | `str` | The exact text that will be spoken by the TTS engine. | `"Linear regression helps us find the line of best fit."` |
| `visuals` | `List[str]` | The specific objects required on screen. | `["Coordinate plane", "Scatter points"]` |
| `animations` | `List[str]` | The actions happening to those objects. | `["Draw regression line"]` |
| `duration` | `int` | Estimated time in seconds. | `5` |

*(Note: While parsing, the script maps LLM outputs like `learning_objective` to `goal`, and `objects` to `visuals` to ensure state compatibility).*

---

## 6. Script Generation Logic

- **Narration Generation:** The LLM expands the short concepts into conversational, educational prose designed for speech.
- **Scene Division:** The agent strictly follows the scene boundaries dictated by the `scene_plans`. It maps one storyboard scene to one planned scene.
- **Visual & Animation Instructions:** The agent is forced by the prompt to list discrete visual elements (`objects`) and actions (`animations`). This acts as an intermediate bridge between abstract concepts and concrete Manim code.
- **Database Persistence:** As soon as the storyboard is validated, the node connects to Supabase and calls `create_scene()` for each storyboard item. It stores the `narration_script` and a formatted string of the `visual_plan` directly in the database so the frontend can display them immediately.
- **Status Update:** The node updates the video status in the database to `WAITING_APPROVAL` to signal the frontend.

---

## 7. Relationship with Planner

The Script Agent acts as the direct executor of the Planner's blueprint.
- **Consumed Fields:** It uses `scene_number`, `title`, and `key_concepts` from the Planner's output to guide the narrative generation.
- **Ignored Fields:** It largely ignores the `duration_seconds` from the Planner, generating its own duration based on the narration length.
- **Modification Power:** The Script Agent **cannot** modify the Planner's decisions. It cannot add, remove, or reorder scenes. It must provide exactly the content requested by the `Teaching Sequence` input.

---

## 8. Relationship with Code Agent

The Script Agent **does not** communicate directly with the Code Agent (Manim Generator). 

1. The `storyboards` state is consumed by the **Scene JSON Generator**.
2. The Scene JSON Generator translates the conversational `visuals` and `animations` into a strict, declarative UI schema (`SceneJSON`).
3. The **Code Agent** receives this semantic JSON, completely decoupled from the English narration.
4. The narration text itself bypasses the code generation entirely. It is saved in the database and later fetched by the **Stitcher** to generate the `.mp3` TTS file using `edge-tts`.

---

## 9. Validation

- **Parsing:** Uses `_parse_json_response()` to strip markdown backticks and clean unescaped backslashes.
- **Validation Rule:** The agent verifies that the `objects` (visuals) and `animations` arrays are not empty. A scene with narration but no visuals violates the core requirement of the platform.
- **Retries:** If the arrays are empty, the agent sets `valid = False`, appends a strict error message (`"ERROR: A scene was generated with empty objects or animations..."`), and invokes the LLM again. It will try up to 3 times before raising a fatal exception.

---

## 10. Important Files

- **`backend/app/agents/nodes/storyboard_agent.py`**: The LangGraph node that invokes the LLM, validates the output arrays, updates the `AgentState`, and syncs the initial records to Supabase.
- **`backend/app/agents/prompts/storyboard_prompts.py`**: Contains the system rules for pedagogical visual storytelling and the prompt assembly logic.
- **`backend/app/agents/state.py`**: Defines the `StoryboardScene` dictionary schema.

---

## 11. Example Walkthrough

1. **Input:** The LangGraph state arrives with `scene_plans` containing: `Scene 1: Introduction (Concepts: gradient descent)`.
2. **Prompt Assembly:** The `storyboard_agent` combines this plan with the `visualization_strategy` (e.g., `physics_motion`) into a string.
3. **LLM Invocation:** The model generates the JSON:
   ```json
   {
     "scene_number": 1,
     "learning_objective": "Explain gradient descent",
     "narration": "Imagine a ball rolling down a hill. Gradient descent works exactly like this.",
     "visual_goal": "Show a ball rolling down a 2D curve",
     "objects": ["2D curve", "Red ball", "Tangent line"],
     "animations": ["Draw curve", "Fade in ball", "Animate ball moving down curve"],
     "duration": 8
   }
   ```
4. **Validation:** The node checks that `objects` and `animations` contain items. It passes.
5. **Database Sync:** The node writes the `narration` and a concatenated string of the visual goals into the Supabase `scenes` table. It updates the video status to `WAITING_APPROVAL`.
6. **State Update:** The node returns `{"storyboards": [...]}` which gets appended to the `AgentState`. The workflow then pauses for human review.
